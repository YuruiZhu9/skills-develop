# Skill Router

这是一个“显式技能路由器”，不是所有任务的通用前置拦截器。

它的职责是帮助你在 **Claude 当前环境中真实可用的 skills** 之间做选择、比较、组合和去歧义。它不负责替代领域 skill，也不应该截走本来就应直接进入某个领域 skill 的任务。

## 何时使用

只在以下情况使用本 skill：

- 用户明确问“该用哪个 skill”
- 用户明确说“帮我选 skill”或“推荐一个 skill”
- 用户明确说“列出可用 skills”或“看看有哪些 skills”
- 用户明确说“组合多个 skills”或“多个 skill 怎么配合”
- 一个任务明显跨多个 skill，且没有单一明显赢家
- 需要比较两个或多个 skill 的适配性、边界或配合方式

## 何时不要使用

出现以下情况时，不要触发本 skill：

- 用户已经明确点名某个 skill
- 用户已经用 `/skill-name` 形式明确调用某个 skill
- 某个流程型 skill 明显应该先介入，例如 `brainstorming`、`systematic-debugging`、`test-driven-development`、`writing-plans`
- 某个领域 skill 已经明显就是正确答案，例如 `pdf`、`docx`、`xlsx`、`frontend-design`
- 只是普通的单领域任务，没有真正的 skill 选择问题
- 你只是因为用户说了 `create`、`build`、`generate` 就想默认触发路由

## 核心原则

### 1. 轻量路由，不要制造额外摩擦

- 不要先向用户发起“是否启动 skill-router”的额外确认
- 先用一句话说明你正在做 skill 路由，然后直接进入判断
- 如果结论非常明确，直接推荐并执行，不要把简单事情复杂化

### 2. 优先使用当前会话中真实可见的 skills

优先级如下：

1. 当前会话系统消息中已经提供的可用 skill 列表
2. 当前会话中已知的 skill 描述与触发条件
3. 仅在确有必要时，再读取候选 skill 的 `SKILL.md`
4. 只有当环境没有直接提供可用列表时，才回退到扫描本地目录

不要虚构 skill 名称、能力或路径。你展示、比较、推荐的 skill，必须来自当前环境中真实存在且真实可用的 skill。

### 3. 先判“直接调用”，再判“是否路由”

路由顺序如下：

1. 用户是否已经点名某个 skill 或直接使用 `/skill-name`
2. 是否存在必须先触发的流程型 skill
3. 是否存在单一明确优先的领域 skill
4. 只有在第 1 到第 3 步都不能直接定案时，才进入多 skill 路由

### 4. 最小化读取范围

- 做清点时，优先依赖当前会话已提供的 skill 列表与描述
- 做定向推荐时，只读取 shortlist 中最可能相关的几个 skill
- 只有在你决定实际使用某个 skill 时，才读取它的完整 `SKILL.md`
- 不要为了一个局部问题扫描全部 skill 正文

### 5. 同名 skill 必须去歧义

如果发现多个 skill 拥有相同名称或近似名称，不要把它们当成同一个：

- 用“名称 + 路径/命名空间 + 描述摘要”来区分
- 推荐时明确写出你选择的是哪一个版本
- 如果当前会话已经明确给出了优先版本，优先跟随当前会话

### 6. 路由完成后要真正执行

一旦路由结论明确：

- 如果已知该 skill 可直接调用，就使用 `Skill` 工具调用它
- 如果需要先理解细则，再读取被选中 skill 的完整 `SKILL.md`
- 不要停留在“推荐完就结束”，除非用户只是在询问选择建议

## 工作流

先判断当前路由任务属于哪一类：

- `A. 盘点当前可用 skills`
- `B. 为具体任务选择单个 skill`
- `C. 为具体任务组合多个 skills`
- `D. 比较冲突 skill 或重名 skill`

根据类型进入相应流程。

## A. 盘点当前可用 skills

当用户只是想知道“有哪些 skills”“当前能用哪些 skill”时，使用这个流程。

### 步骤

1. 优先使用当前会话系统消息里可见的 skill 列表
2. 按名称和描述做快速分类
3. 仅当当前环境未提供列表时，才扫描本地 skill 目录
4. 如果扫描本地目录，优先读取 frontmatter 或最小必要元数据
5. 按真实发现到的 skill 做分类，不要引用并不存在的示例 skill

### 输出要求

输出应包含：

- 总数
- 按类别分组后的清单
- 可能的重名 skill
- 如有必要，说明“本地目录清点结果”和“当前会话实际可用 skills”之间可能存在差异

### 输出模板

可以采用这种结构：

```text
可用 Skills 概览

总计：N

前端与设计
- frontend-design: 用于高质量前端界面设计
- ui-ux-pro-max: 用于 UI/UX 设计与评审

文档与办公
- pdf: 处理 PDF 文件
- docx: 处理 Word 文档

流程与工程
- brainstorming: 在创意实现前做设计澄清
- systematic-debugging: 系统化排查 bug

重名项
- ui-ux-pro-max
  - 版本 A: ...
  - 版本 B: ...
```

## B. 为具体任务选择单个 skill

当用户有明确任务，但不知道该用哪个 skill 时，使用这个流程。

### 步骤

1. 先理解任务目标、交付物和约束
2. 判断是否有流程型 skill 必须优先介入
3. 如果没有，再筛出 2 到 5 个最相关的候选 skill
4. 先看当前会话中的描述，必要时再读 shortlist 的完整 `SKILL.md`
5. 如果存在明显赢家，直接推荐并执行
6. 只有在存在真实权衡且你无法客观定案时，才把选择权交给用户

### 评估标准

按以下顺序判断：

1. 用户是否已经点名某个 skill
2. 是否存在必须优先的流程型 skill
3. 任务匹配度是否最强
4. 输出质量要求是否最契合
5. 技术栈和环境是否匹配
6. 完成目标的复杂度是否最低
7. 用户是否有已知偏好

### 输出模板

如果存在明显赢家，直接这样说：

```text
推荐使用：`[skill-name]`

原因：
1. 它和当前任务的匹配度最高
2. 它能直接覆盖你要的交付物
3. 不需要额外引入第二个 skill 才能完成

下一步：调用 `[skill-name]`，并按其流程执行。
```

## C. 为具体任务组合多个 skills

当一个任务确实需要多个 skill 协作时，使用这个流程。

### 先判断是否真的需要组合

只有在以下情况才组合：

- 一个 skill 负责主流程，另一个 skill 负责明确的补充能力
- 不同 skill 分别覆盖不同交付物
- 单一 skill 明显覆盖不全，但多个 skill 之间不存在硬冲突

如果单一 skill 已经足够，不要为了“看起来更强”而强行组合。

### 组合方式

优先使用以下结构：

- `主 skill + 辅助 skill`
- `流程 skill + 领域 skill`

不要默认采用“多个 skill 平铺并列”的方式，因为这会增加冲突和上下文噪音。

### 输出模板

```text
建议采用：主辅组合

主 skill：`[primary-skill]`
辅助 skill：`[aux-skill]`

原因：
- `[primary-skill]` 负责主流程和主要交付物
- `[aux-skill]` 只补充其中的某个明确环节

执行顺序：
1. 先调用或读取 `[primary-skill]`
2. 再读取 `[aux-skill]` 的相关部分
3. 以主 skill 为主线执行，只吸收辅助 skill 的必要技术或规范
```

## D. 比较冲突 skill 或重名 skill

当两个 skill 看起来都适用，或者同名 skill 出现多个版本时，使用这个流程。

### 冲突处理

如果两个 skill 的建议互相冲突：

- 先看是否一个是流程 skill，另一个是领域 skill
- 再看哪个更贴近当前任务的真实交付物
- 再看哪个对当前环境更合适
- 如果你能客观判断，直接做决定并简短解释
- 如果确实是价值取舍而不是信息不足，再把差异讲给用户

### 重名处理

如果同名 skill 有多个来源：

- 在输出中标注路径、命名空间或来源
- 不要只写 skill 名称
- 选择时优先使用当前会话实际可调用的那个版本

### 输出模板

```text
发现同名或冲突 skill：

- `[skill-a]` - 来源 A - 偏向 ...
- `[skill-b]` - 来源 B - 偏向 ...

本次建议选择：来源 A 对应的版本

原因：
1. 它与当前任务更贴合
2. 当前会话更容易直接调用该版本
3. 另一个版本没有提供额外决定性优势
```

## 执行阶段

一旦路由结论已经明确：

- 直接调用被选中的 skill，或读取其完整 `SKILL.md` 后调用
- 如果是组合方案，只读取主 skill 全文和辅助 skill 的必要部分
- 明确告诉用户你接下来要使用哪个 skill 或哪组 skill
- 然后真正进入执行，不要停留在路由讨论里

## 重要提醒

- 本 skill 不是默认入口
- 不要拦截本来就该直接进入领域 skill 的任务
- 不要仅因任务“看起来复杂”就启动路由
- 不要虚构 skill 名称或能力
- 不要为了盘点而读取全部 skill 正文
- 不要把重名 skill 混为一谈
- 如果结论已经很清楚，就快速推荐并执行
